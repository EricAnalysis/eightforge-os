# Contract Rate Assembler and Description Categorization Audit

**Audit date:** 2026-06-20  
**Verdict:** Pass with significant concerns. The current code can match a code-less `$6.90/CY` vegetative ROW-to-DMS line when description, unit, rate, and a loaded pricing-family row are present, but the assembler and validator maintain divergent taxonomies and the validator can substitute a filtered assembler subset for the persisted schedule.

## 1. Contract Rate Assembler Location

The operator-facing Contract Rate Assembler is `assembleContractPricingRows` in `lib/contracts/contractPricingAssembly.ts:1525-1685`. It merges typed rows, canonical rows, and persisted `ContractRateScheduleRow` values; normalizes OCR text, category, route, distance, unit, rate, description, confidence, source kind, source quality, and anchor; then deduplicates, groups, limits, and sorts the result (`lib/contracts/contractPricingAssembly.ts:1525-1553`, `lib/contracts/contractPricingAssembly.ts:1566-1587`, `lib/contracts/contractPricingAssembly.ts:1615-1685`).

The upstream rate-row builder is `buildContractRateScheduleRows` in `lib/contracts/contractRateScheduleRows.ts:628-663`. Its priority is extracted Exhibit A tables plus targeted text recovery, then typed rate-table facts, then parsed source-entry text (`lib/contracts/contractRateScheduleRows.ts:638-663`). `analyzeContractIntelligence` calls this builder and persists the result as `contract_analysis.rate_schedule_rows` (`lib/contracts/analyzeContractIntelligence.ts:1016-1046`).

The UI call chain is:

`buildDocumentIntelligenceViewModel` → `assembleContractPricingRows(executionTrace.contract_analysis.rate_schedule_rows, { canonicalRows, typedRows })` (`lib/documentIntelligenceViewModel.ts:6068-6077`) → operator-facing contract rate rows.

The validator call chain is:

`loadProjectValidatorInput` → precedence/relationship loading → `buildDocumentIdsByFamily` → `buildContractValidationContext` → `buildFactLookups` → `buildRateScheduleItems` → `assembleContractPricingRows` → `normalizeRateScheduleItem` → `indexRateScheduleItemsByCanonicalKeys` → `matchRateScheduleItemForInvoiceLine` → `evaluateCrossDocumentRateVerification` → finding/summary (`lib/validator/projectValidator.ts:2148-2170`, `lib/validator/projectValidator.ts:2214-2232`, `lib/validator/projectValidator.ts:1422-1476`, `lib/validator/rulePacks/crossDocumentRateVerification.ts:732-821`).

There is no single taxonomy shared by all of those stages. The assembler has eight allowed display categories (`lib/contracts/contractPricingAssembly.ts:60-82`); `rateTaxonomy.ts` has four canonical categories (`lib/validator/rateTaxonomy.ts:26-109`); and operational matching independently infers five categories (`lib/validator/billingKeys.ts:598-633`).

## 3. Description Categorization Map

There are no explicit parent/child category relationships. Categories are flat strings; aliases are matched by exact, substring, or all-token inclusion, with confidence weights (`lib/validator/rateTaxonomy.ts:120-165`, `lib/validator/rateTaxonomy.ts:202-267`). The confidence threshold is `0.68`; existing canonical values default to `0.95`, source category matches score `0.82-0.96`, descriptor matches `0.74-0.88`, and combined evidence adds up to `0.06` (`lib/validator/rateTaxonomy.ts:3`, `lib/validator/rateTaxonomy.ts:174-187`, `lib/validator/rateTaxonomy.ts:202-235`, `lib/validator/rateTaxonomy.ts:270-278`).

Normalization is deterministic but layered. Basic billing normalization lowercases, removes all non-alphanumeric punctuation, collapses whitespace, and does no stemming or general abbreviation/synonym expansion (`lib/validator/billingKeys.ts:31-40`). Operational normalization additionally expands `&` to `and`, `FDS` to `final disposal`, and two OCR artifacts, then uses token-level Levenshtein distance of one or two edits (`lib/validator/billingKeys.ts:481-534`). Contract assembly has a separate OCR cleanup map for arrows, `ROW/DMS/FDS`, `Milesfrom`, and common OCR corruptions (`lib/contracts/contractPricingAssembly.ts:177-203`). Unit cleanup is also separate between assembler and matcher (`lib/contracts/contractPricingAssembly.ts:206-229`, `lib/validator/billingKeys.ts:656-681`).

## 4. Relationship Graph

```text
Document
  -> OCR/PDF tables + text evidence
  -> ContractRateScheduleRow (page + source_anchor_ids)
  -> contract_analysis.rate_schedule_rows / normalized rate_table facts
  -> Contract Pricing Assembler (display category, route, distance, unit, rate, quality)
  -> RateScheduleItem (document id + record id + billing/description keys)
  -> invoice-line candidate matching
       exact billing key
       -> exact description key
       -> operational fallback: category + unit + exact rate + token/route/distance
  -> selected contract rate row
  -> validation unit
  -> validator finding or resolved match
```

The document→table→row relationship is explicit through page and source anchors (`lib/contracts/contractRateScheduleRows.ts:420-435`). Contract-family eligibility is explicit at document-list construction: attached pricing sources are placed in pricing truth, and rate facts are limited to contract identity plus pricing IDs (`lib/validator/projectValidator.ts:1064-1093`, `lib/validator/projectValidator.ts:1685-1699`).

The family relationship is not passed into or scored by the row matcher. Once `rateScheduleItems` is built, matching operates over one flat index and has no document-family, precedence, relationship-type, or temporal-authority field (`lib/validator/billingKeys.ts:286-315`, `lib/validator/billingKeys.ts:684-717`). Thus “belongs to the governing family” is an upstream inclusion filter, not positive per-candidate evidence.

The selected row relationship is explicit by `source_document_id` and `record_id` in the validation unit (`lib/validator/rulePacks/crossDocumentRateVerification.ts:787-796`). The reason is only partially explicit: the unit stores a high-level classification reason, but not matcher `match_reason`, candidate score, token matches, unit/rate/category reasons, or rejected alternatives (`lib/validator/rulePacks/crossDocumentRateVerification.ts:766-797`).

## 5. Goodlettsville Match Trace

The bundled real Goodlettsville fixture exists at `lib/contracts/__fixtures__/goodlettsville_price_sheet.pdf`, and its golden OCR test expects five page-two rows. Its vegetative loading/hauling row is `$27.00/CY`, not `$6.90/CY` (`lib/extraction/pdf/goodlettsvillePriceSheet.test.ts:75-104`). Therefore the exact source document for the stated `$6.90/CY, 43,894 CY` blocker is not this bundled PDF. The repository does contain that exact rate/quantity scenario as a synthetic cross-document test (`lib/validator/rulePacks/crossDocumentRateVerification.test.ts:244-282`).

For the synthetic line `Vegetative Collect Remove Haul Unincorporated Neighborhoods ROW to DMS 0 to 15`, `$6.90/CY`, quantity `43,894`:

1. Basic description normalization produces alphanumeric lowercase tokens; no code is required (`lib/validator/billingKeys.ts:31-40`, `lib/validator/billingKeys.ts:94-112`).
2. Invoice category resolves to `vegetative_removal` from description/material aliases (`lib/validator/rulePacks/crossDocumentRateVerification.ts:196-235`, `lib/validator/rateTaxonomy.ts:76-96`).
3. Exact code matching fails harmlessly if the contract has no code; the matcher proceeds from empty exact candidates to operational fallback (`lib/validator/billingKeys.ts:321-347`, `lib/validator/billingKeys.ts:816-851`).
4. Operational fallback infers vegetative category from vegetative/ROW/DMS, normalizes CY, requires the contract rate to equal `$6.90` within `$0.01`, scores description-token coverage, and adds route/distance bonuses (`lib/validator/billingKeys.ts:556-595`, `lib/validator/billingKeys.ts:624-704`).
5. If one best candidate remains, it becomes the contract row; the test confirms `contract_rate_found=true`, rate `6.9`, and a resolved match (`lib/validator/rulePacks/crossDocumentRateVerification.test.ts:275-282`).
6. Quantity `43,894` does not add matching confidence. It is context only; support quantity is calculated after the row match (`lib/validator/rulePacks/crossDocumentRateVerification.ts:664-677`, `lib/validator/rulePacks/crossDocumentRateVerification.ts:755-764`).

Confidence is lost at these exact gates:

- The pricing row never enters `rateScheduleItems` because the price sheet is not classified/related into contract identity or pricing truth (`lib/validator/projectValidator.ts:1064-1093`, `lib/validator/projectValidator.ts:1685-1699`).
- The assembler drops or limits a persisted row before validator normalization; the validator uses assembled rows wholesale whenever at least one survives (`lib/contracts/contractPricingAssembly.ts:1301-1391`, `lib/validator/projectValidator.ts:1422-1441`).
- Unit fields are absent or malformed. Missing units are permissive, but conflicting resolved units reject the candidate (`lib/validator/billingKeys.ts:656-690`).
- Rate differs by more than `$0.01`; operational fallback then rejects the row instead of returning a rate-mismatch candidate (`lib/validator/billingKeys.ts:692-704`). The corresponding regression test expects a missing-contract-rate finding, not a rate-mismatch finding (`lib/validator/rulePacks/crossDocumentRateVerification.test.ts:321-356`).
- Description token coverage is below `0.45`, or below `0.75` for `needs_review` contract rows (`lib/validator/billingKeys.ts:417-420`, `lib/validator/billingKeys.ts:700-704`).
- Two distinct candidates have exactly equal score; the matcher returns ambiguous/null unless they collapse to the same description/category/unit/rate slot (`lib/validator/billingKeys.ts:720-766`).
- Invoice category is unresolved, in which case comparison stops at `needs_review` even if a row was selected (`lib/validator/rulePacks/crossDocumentRateVerification.ts:438-451`).

The current focused suite passes: 7 files and 127 tests, including the real Goodlettsville OCR fixture. The fixture logs a missing OpenAI API key for optional vision supplementation but still passes through deterministic OCR recovery; this verifies extraction fallback, not an end-to-end project-family match.

## 6. Add / Combine / Remove Recommendations

### Add

- Add canonical categories for `management_reduction`, `final_disposal`, `equipment`, `personnel`/`labor`, and `specialty_removal`; they already exist in assembler or operational inference but not in the shared taxonomy (`lib/contracts/contractPricingAssembly.ts:60-69`, `lib/validator/billingKeys.ts:598-633`, `lib/validator/rateTaxonomy.ts:26-109`). **Needs operator confirmation:** whether personnel and labor are one category or parent/child services.
- Add aliases for `ROW`, `right of way`, `right-of-way`, `DMS`, `debris management site`, `FDS`, `final disposal site`, `CY`, `CYD`, `cubic yard(s)`, loading/hauling, debris-site management, reduction, air-curtain burning, open burning, compaction, and pass-through/tipping. Existing support is fragmented across assembler, matcher, and fixture-only normalizers (`lib/contracts/contractPricingAssembly.ts:177-229`, `lib/validator/billingKeys.ts:481-501`, `lib/extraction/pdf/goodlettsvillePriceSheet.test.ts:31-56`).
- Add explicit service dimensions alongside category: material, action, origin, destination, distance band, and unit. The assembler already derives route/distance but the canonical taxonomy remains one flat label (`lib/contracts/contractPricingAssembly.ts:383-430`, `lib/validator/rateTaxonomy.ts:12-24`).
- Add per-match provenance: `match_reason`, score, description score, normalized unit, rate delta, inferred categories, family eligibility, candidate count, ambiguity, and rejected candidates. The matcher computes several of these but the validation unit omits them (`lib/validator/billingKeys.ts:437-449`, `lib/validator/billingKeys.ts:684-717`, `lib/validator/rulePacks/crossDocumentRateVerification.ts:766-797`).
- Add positive candidate-level contract-family evidence rather than relying only on upstream filtering (`lib/validator/projectValidator.ts:1685-1699`, `lib/validator/billingKeys.ts:286-315`).
- Add quantity reconciliation as separate corroborating evidence, not a primary rate-row identity key (`lib/validator/rulePacks/crossDocumentRateVerification.ts:755-764`).

### Combine

- Combine the assembler display-category map, shared canonical taxonomy, and operational inference behind one canonical registry with presentation labels and aliases. Today they are independent (`lib/contracts/contractPricingAssembly.ts:60-123`, `lib/validator/rateTaxonomy.ts:26-109`, `lib/validator/billingKeys.ts:598-633`).
- Combine duplicate normalization of punctuation, OCR aliases, unit aliases, and route/distance tokens into shared deterministic helpers while preserving raw source values (`lib/validator/billingKeys.ts:31-40`, `lib/validator/billingKeys.ts:481-501`, `lib/contracts/contractPricingAssembly.ts:177-229`).
- Combine `vegetative`, `vegetation`, and `vegetative debris` as aliases of the material dimension, but do **not** combine collection/haul, management/reduction, and final disposal merely because all handle vegetative material (`lib/validator/rateTaxonomy.ts:76-96`, `lib/contracts/contractPricingAssembly.ts:60-65`).

### Remove / Deprecate

- Deprecate `mulch` as an unconditional source-category alias for `vegetative_removal`; it is too broad across hauling, reduction, and disposal. Keep it as a material alias requiring action/route context (`lib/validator/rateTaxonomy.ts:76-96`). **Needs operator confirmation.**
- Deprecate route-only vegetative inference `(ROW && DMS)` and standalone `rural`/`unincorporated` inference; C&D can use the same route/scope (`lib/validator/billingKeys.ts:624-630`, `lib/contracts/contractPricingAssembly.ts:344-347`).
- Deprecate standalone `hazardous` as sufficient tree evidence; require tree/limb/stump/hanger context (`lib/validator/billingKeys.ts:618-620`).
- Remove no category now: repository usage proves all eight assembler categories are expected and capped, while no usage proof supports deletion (`lib/contracts/contractPricingAssembly.ts:60-82`, `lib/contracts/contractPricingAssembly.ts:1342-1372`).
- Deprecate hard-coded page/category/count policy as a general validator input. It is Williamson-specific (`lib/contracts/contractPricingAssembly.ts:73-94`, `lib/contracts/contractPricingAssembly.ts:1342-1372`) but is called on generic persisted contract rows by the validator (`lib/validator/projectValidator.ts:1422-1441`).

## 7. Failure Modes

1. **Partial schedule loss:** if the assembler recognizes at least one persisted row, the validator uses the assembled subset instead of all persisted rows; dropped categories do not fall back individually (`lib/validator/projectValidator.ts:1422-1441`).
2. **Project-specific caps truncate valid rows:** category counts are hard-coded to 9/5/5/8/10/16/9/53 and rows beyond those limits are sliced away (`lib/contracts/contractPricingAssembly.ts:73-82`, `lib/contracts/contractPricingAssembly.ts:1359-1372`).
3. **Page-specific category rejection:** pages 8-11 encode expected Williamson categories, which is unsafe as a general contract rule (`lib/contracts/contractPricingAssembly.ts:84-94`, `lib/contracts/contractPricingAssembly.ts:1311-1316`).
4. **Canonical-category loss during validator assembly:** the mapping from assembled rows to validator rows includes display category but not original canonical category/confidence; normalization must re-infer it (`lib/validator/projectValidator.ts:1422-1440`, `lib/validator/projectValidator.ts:1354-1359`).
5. **Taxonomy split:** management/reduction, final disposal, equipment, personnel, and specialty removal can be displayed but remain unresolved or miscategorized in shared validation (`lib/contracts/contractPricingAssembly.ts:60-69`, `lib/validator/rateTaxonomy.ts:26-109`).
6. **Wrong combination:** grinding/chipping vegetative work maps to `vegetative_removal`, combining reduction with collection/haul (`lib/validator/rateTaxonomy.ts:83-96`, `lib/contracts/contractPricingAssembly.ts:499-506`).
7. **Wrong split:** aliases with minor unlisted wording may become unresolved because taxonomy matching is exact/contains/all-token, not phrase-aware beyond the listed aliases (`lib/validator/rateTaxonomy.ts:120-165`).
8. **False positive from same rate:** guarded by category, unit, and description threshold, but permissive missing category/unit values can still admit a same-rate candidate if token score passes (`lib/validator/billingKeys.ts:636-681`, `lib/validator/billingKeys.ts:684-704`).
9. **False negative becomes “missing rate” rather than “rate mismatch”:** operational matching requires exact rate before selecting a candidate (`lib/validator/billingKeys.ts:692-704`), so a semantically correct row with a changed invoice rate is reported as no confident contract row (`lib/validator/rulePacks/crossDocumentRateVerification.test.ts:321-356`).
10. **Code alias dominance on exact candidates:** exact billing-key lookup is attempted first; a wrong code can select a candidate before semantic fallback, although rate/description/unit checks now guard code-less/exact-description paths (`lib/validator/billingKeys.ts:321-347`, `lib/validator/billingKeys.ts:816-841`).
11. **Ambiguity lacks operator detail:** equal-score distinct rows return null, but the finding does not show the candidates or scores (`lib/validator/billingKeys.ts:720-766`, `lib/validator/rulePacks/crossDocumentRateVerification.ts:687-713`).
12. **Family relationship silently disappears:** precedence loading catches all errors and substitutes empty relationships/families, which can exclude the attached pricing sheet without surfacing the relationship-load failure here (`lib/validator/projectValidator.ts:2148-2163`).
13. **Only first contract identity drives contract analysis:** `buildContractValidationContext` selects `contract_identity[0]`; related pricing rows are later added from facts, but related documents are not passed into synthetic contract analysis (`lib/validator/projectValidator.ts:1614-1649`).
14. **Ticket over-grouping:** support falls back from invoice-scoped matches to any project row with the same category, allowing unrelated invoices to provide support (`lib/validator/rulePacks/crossDocumentRateVerification.ts:375-418`).
15. **Existing canonical values are trusted without registry validation:** any non-empty existing category is normalized and accepted with default confidence `0.95`, even when absent from `CATEGORY_RULES` (`lib/validator/rateTaxonomy.ts:174-187`).
16. **Evidence reason gap:** source document/row provenance survives, but the decisive normalized tokens, score, match reason, and family edge do not (`lib/validator/rulePacks/crossDocumentRateVerification.ts:766-797`).
